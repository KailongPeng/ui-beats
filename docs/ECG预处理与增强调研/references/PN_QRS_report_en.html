Section 00

Background & Technical Principles

Problem

Why existing QRS detectors fall short

Hospital ECGs are clean — traditional methods (Pan-Tompkins) are sufficient.
But wearable devices recording long-term ECG face:
  • Motion artifacts, baseline wander, poor electrode contact
  • Noise segments that look like QRS — detectors can't tell
  • Traditional methods give yes/no — no confidence score
Innovation

PN-QRS core answer

The name Positive-Negative comes from its key mechanism:
For every signal window, create an "optimistic clone" (φ⁺) and a "pessimistic clone" (φ⁻). Their disagreement estimates uncertainty, enabling three output tiers:
● This window is noise — discard entirely ● This beat is morphologically ambiguous — flag ● Precise R-peak location — confident output

Two Uncertainties: The System's Core Distinction

UE
Epistemic Uncertainty
"The model has never seen this type of signal"
The model was trained only on ECG. When given PPG, heart sounds, or pure noise, the positive/negative clones strongly disagree → U_E spikes.

U_E high → the entire 10s window is non-ECG — discard it.
Formula
UE = KL(ŷ⁺ ‖ ŷ) + KL(ŷ⁻ ‖ ŷ)
Threshold α = 0.1 (paper) / 2.45 (local calibration)
Measured: pure noise U_E ≈ 3–4, clean ECG ≈ 1.4 — clearly separable
UA
Aleatoric Uncertainty
"The data itself is ambiguous — even experts would hesitate"
The signal IS valid ECG, but the beat morphology is unusual (PVC, LBBB) — even a cardiologist would double-check.

U_A high → the R-peak is flagged as ambiguous; the signal itself is kept.
Formula
UA = −[ŷ⁺ log ŷ⁺ + (1−ŷ⁻) log(1−ŷ⁻)]
Threshold β = 0.12, averaged over ±7 frames around each R-peak
Unlike U_E: does not discard the window — only flags that beat for downstream analysis

All datasets are in WFDB format (.dat + .hea + .atr) with varying sampling rates and record lengths. PN-QRS operates in a 50 Hz internal space — all data must go through a unified preprocessing pipeline before inference.
